# Tool layer reference

The agent's action surface (deliverable 2). Generated from `src/store_agent/tools.py` via `uv run python -m store_agent.tools` â€” regenerate after any tool change. Parameters marked `?` are optional.

| Tool | Parameters | Description |
|---|---|---|
| `search_products` | `query`, `color`?, `size`? | Resolve a product description to concrete SKUs with stock and list price. Always call this before selling, returning, restocking, or discounting a product. If more than one variant matches and the user did not fully specify, ask the user which one they mean instead of picking. |
| `lookup_customer` | `query` | Find a customer by (partial) name, email, or id. Sales without a customer are walk-ins â€” never attach a guessed customer. |
| `ring_up_sale` | `items`, `date`, `customer_id`?, `order_discount_pct`?, `payment_method`? | Record a sale â€” required whenever the user says ring up / sell / record, even if they also ask for the price (the result includes it). Checks stock (refuses the whole sale if any item is short), applies any active promotion for the sale date, decrements inventory, and returns the authoritative prices and order total. Omit customer_id for walk-ins. |
| `get_order` | `order_id` | Fetch one order with its lines, per-unit prices actually paid (after the order-level discount), and any returns already recorded against it. Use before processing a return. |
| `list_orders` | `customer_id`?, `start_date`?, `end_date`? | List orders, optionally filtered by customer and/or date range. |
| `process_return` | `order_id`, `sku`, `quantity`, `condition`, `date`? | Return units from a past order. Refunds the price actually paid on that order (never the current list price). condition='good' restocks the units; 'damaged' does not â€” this materially changes the outcome, so if the user hasn't stated the condition, ask before calling this tool; never default to 'good'. Rejects returning more than was sold minus already returned. Pass the quantity the user asked for â€” if it exceeds what remains returnable, do NOT process a smaller quantity instead; report the limit and ask the user first. For a bulk 'return everything from order X' request, call get_order first, then call this tool once per line using the skus and quantities it shows â€” that's a complete, unambiguous spec, so proceed without asking again (still ask about condition if unstated). |
| `restock_below_reorder` | `date`? | Scan all inventory and create purchase orders (one per supplier) for every SKU at or below its reorder point, ordering its reorder quantity from the cheapest supplier that delivers within 10 days. Returns the POs plus the per-SKU supplier decisions. |
| `create_purchase_order` | `supplier_id`, `items`, `date`? | Create a purchase order for specific items from a specific supplier (use restock_below_reorder for rule-based restocking). supplier_id accepts an id ('SUP-NW') or an unambiguous supplier name ('Northwind'). Unit costs come from the supplier's catalog. If the user describes a PO as if it already exists (e.g. 'a PO for 50 totes from Northwind is open') and list_purchase_orders shows no such PO, call this tool immediately with exactly the details given, then continue (e.g. receive against it) â€” do not stop to ask the user to confirm creating it first; they already told you what to create. |
| `list_purchase_orders` | `status`? | List purchase orders with their lines and received quantities. Filter by status 'open' or 'received'. |
| `receive_purchase_order` | `po_id`, `receipts`?, `date`? | Book a delivery against an open purchase order: adds the units to inventory. Pass receipts for a partial delivery (e.g. 40 of 50 arrived); omit receipts to receive everything outstanding. The PO closes automatically when complete. If the user's phrasing doesn't identify which PO (e.g. 'the shipment arrived' while more than one PO is open), call list_purchase_orders and ask which one â€” do not guess by receiving every open PO unless the user clearly means all of them. |
| `create_promotion` | `description`, `percent_off`, `scope_type`, `scope_ref`, `start_date`, `end_date` | Create a REAL, persistent percent-off promotion. Only call this when the user is actually instructing you to set up or apply a promotion. Never call it just to answer a hypothetical 'what would X cost if...' question â€” use simulate_discount_price for those instead. scope_type='product' with scope_ref = product_id covers ALL variants of one product â€” e.g. 'all hoodies' means P-HOOD, NOT the apparel category. Use scope_type='category' (scope_ref 'apparel' or 'goods') only when the user names a whole category. Start/end dates are inclusive. Promotions never change past sales; overlapping promotions don't stack â€” the lower price wins. |
| `get_price_quote` | `sku`, `date`? | The effective per-unit price of a SKU on a given date, with any promotion applied. ONLY for hypothetical 'what would X cost' questions â€” it records nothing. If the user asked to ring up / sell / record a sale, use ring_up_sale instead; its result already includes the price to report. |
| `simulate_discount_price` | `sku`, `percent_off` | Compute a HYPOTHETICAL price for a SKU under an arbitrary percent-off, without creating or changing anything â€” records nothing. Use this for 'what would X cost if we put it on Y% off' questions. It answers only the hypothetical asked (it does not layer on top of other active promotions). To actually set up a discount for real, use create_promotion instead â€” never call create_promotion just to answer a hypothetical. |
| `revenue_report` | `start_date`?, `end_date`? | Gross revenue (dollars actually paid on orders in the period), refunds issued in the period, and net revenue. Defaults to last month (May 2026). |
| `top_products_by_margin` | `start_date`?, `end_date`?, `limit`? | Products ranked by margin dollars for a period (margin % included). Margin follows the store rule: revenue from a product's units minus Northwind cost of units that stayed sold; restocked returns count in neither. Defaults to last month (May 2026). |
| `stockout_report` | â€” | Which products are about to stock out: any variant at/below its reorder point, or fewer than 14 days of cover at last-30-days sales velocity. |
| `run_sql` | `query` | Read-only SELECT against the store database for questions no other tool answers. Tables: products(sku, product_id, product_name, category, color, size, retail_price), customers, suppliers, supplier_catalog(supplier_id, product_id, unit_cost, lead_time_days), inventory(sku, on_hand_qty, reorder_point, reorder_qty), orders(order_id, order_date, customer_id, order_discount_pct, payment_method), order_lines(order_id, line_no, sku, quantity, unit_price), returns, promotions, purchase_orders, purchase_order_lines. Prefer the report tools for revenue/margin/stock-out â€” they encode the store's frozen business rules. |
